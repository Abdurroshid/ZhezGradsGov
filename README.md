# Реестр аудита показателей

Портал сбора показателей: каждая организация заполняет **только свою** строку, аудитор
видит **сводный реестр** всех организаций. Frontend — один файл `index.html`. Backend —
**Supabase** (PostgreSQL + Auth + Row-Level Security). Доступ проверяется на сервере, а не
в браузере: организации не могут видеть данные друг друга даже через DevTools.

---

## Что внутри

| Файл | Назначение |
|------|-----------|
| `index.html` | Весь интерфейс (вход, форма организации, панель аудитора). |
| `supabase/schema.sql` | Таблицы, политики безопасности RLS, триггеры, стартовые столбцы. |
| `README.md` | Этот файл. |

Модель данных:

- **profiles** — профиль организации (название, БИН, email, роль `org`/`admin`), 1:1 с `auth.users`.
- **indicators** — столбцы реестра (Бюджет, Код ДПД, Наименование КБК, Ожидаемая на; аудитор может добавлять).
- **submissions** — поданные данные, по одной строке (JSON `values`) на организацию.

## Настройка (≈10 минут)

### 1. Создать проект Supabase
1. Зайти на [supabase.com](https://supabase.com) → **New project**.
2. Задать имя, пароль базы, регион (ближайший — напр. Frankfurt / Singapore). Дождаться готовности (~2 мин).

### 2. Создать таблицы и политики
1. В проекте открыть **SQL Editor → New query**.
2. Вставить целиком содержимое `supabase/schema.sql` и нажать **Run**.
   Скрипт можно запускать повторно — он не ломает существующие данные.

### 3. Прописать ключи в приложении
1. **Project Settings → API**.
2. Скопировать **Project URL** и **anon public** ключ.
3. В `index.html`, вверху `<script>`, вставить их:
   ```js
   const SUPABASE_URL      = 'https://ваш-проект.supabase.co';
   const SUPABASE_ANON_KEY = 'eyJhbGciOi...';   // anon public — публиковать безопасно
   ```
   > anon-ключ **предназначен** для фронтенда. Реальную защиту даёт RLS, а не секретность ключа.
   > Никогда не вставляйте сюда `service_role` ключ.

### 4. Создать аудитора (admin)
1. Открыть приложение, вкладка **Регистрация** — зарегистрировать нужный email (напр. `admin@audit.gov.kz`).
2. В **SQL Editor** выполнить (подставив свой email):
   ```sql
   update public.profiles set role = 'admin'
   where id = (select id from auth.users where email = 'admin@audit.gov.kz');
   ```
3. Войти заново — откроется панель аудитора со сводным реестром.

Все остальные, кто регистрируется, автоматически получают роль `org` и видят только свою форму.

### 5. Подтверждение email (по желанию)
**Authentication → Providers → Email** в Supabase:
- **Confirm email = ON** (по умолчанию): после регистрации приходит письмо со ссылкой; вход — после подтверждения. Рекомендуется для реального запуска.
- **Confirm email = OFF**: вход сразу после регистрации. Удобно для тестирования; аудитор сам заводит организации.

## Публикация (хостинг фронтенда)

`index.html` — статический файл, ему не нужен свой сервер. Любой вариант бесплатен:

- **Netlify / Cloudflare Pages / Vercel** — перетащить папку или подключить репозиторий (auto-deploy при push).
- **GitHub Pages** — Settings → Pages → включить для ветки.

После деплоя добавьте адрес сайта в Supabase: **Authentication → URL Configuration → Site URL / Redirect URLs**, иначе письма подтверждения будут вести не туда.

## Безопасность и ограничения

- **RLS включён на всех таблицах** — сервер сам не отдаёт чужие строки. Это ядро защиты.
- Организация **не может** повысить себе роль до `admin` (запрещено триггером `prevent_role_escalation`).
- **Удаление организации** аудитором убирает её `profile` и `submission`. Сама учётная запись входа
  (`auth.users`) остаётся; при повторном входе приложение обнаружит отсутствие профиля и выйдет из сессии.
  Чтобы удалить и вход полностью — **Authentication → Users → Delete user** в дашборде Supabase.
- Пароли хранит и хеширует Supabase Auth (мы их не видим и не храним).
